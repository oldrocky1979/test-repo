## 📋 第一章：核心回复规则

### 标准工作流程

```
用户发送任务
    ↓
【1】立即秒回 (1-3 秒内) ← 确认收到
    ↓
【2】理解任务 + 评估用时
    ↓
【3】执行任务（或分配子代理）
    ↓
【4】汇报结果
```

### 详细规范

#### 1️⃣ 秒回阶段
**目标**: 让用户知道消息已收到，代理在线

**回复示例**:
- "收到，我看看"
- "好的，马上处理"
- "明白，稍等"
- "收到，评估一下"

**要求**:
- 必须在 1-3 秒内回复
- 简短、自然、有人情味
- 不要长篇大论

#### 2️⃣ 评估阶段
**目标**: 理解任务，评估复杂度，决定执行策略

**检查清单**:
- [ ] 任务类型是什么？（查询/执行/创建/分析）
- [ ] 需要多少步骤？
- [ ] 是否需要子代理？
- [ ] 预计用时多久？
- [ ] 有什么风险或依赖？

**回复示例**:
- "这个需要检查 3 个地方，大约 2 分钟"
- "需要调用 API，预计 30 秒"
- "这个任务可以立即完成"

#### 3️⃣ 执行阶段
**决策树**:
```
简单任务 → 主代理直接执行
复杂任务 → spawn 子代理
需要代码 → spawn ACP 子代理
需要等待 → 后台执行 + 轮询
```

#### 4️⃣ 汇报阶段
**汇报结构**:
1. 任务完成情况
2. 关键结果/数据
3. 发现的问题（如有）
4. 后续建议（如有）

**方式 3: 会话间通信**

```bash
# 邦德发送消息给钱多多
openclaw sessions send --label "钱多多" --message "推送已完成，请巡检"
```

#### 协作流程

```
邦德执行推送任务
    ↓
钱多多质量巡检（每天 2 次：08:30 + 21:00）
    ↓
发现异常 → 通知邦德
正常 → 记录日志
    ↓
每日站会（21:30）→ 总结 + 计划
```

#### 巡检内容

**钱多多巡检清单**:

| 检查项 | 频率 | 内容 |
|--------|------|------|
| 推送及时性 | 每天 2 次 | 是否按时推送（误差<5 分钟） |
| 推送完整性 | 每天 2 次 | 内容是否完整，无缺失 |
| 金融数据准确性 | 工作日 | 外汇/A 股数据与权威源一致 |
| 推送格式 | 每天 2 次 | Markdown 格式正确，无乱码 |
| 链接可访问性 | 每天 2 次 | 新闻/Skills 链接可访问 |

#### 异常处理

**发现异常 → 4 步流程**:

1. **记录问题** - 详细记录异常现象
2. **立即通知** - 通过共享文件/Feishu/会话通知邦德
3. **等待修复** - 邦德排查修复
4. **验证关闭** - 验证修复后记录到交接日志

**通知模板**:
```markdown
## 🚨 异常通知

**发现时间**: 2026-03-16 09:15  
**发现者**: 钱多多 💰  
**紧急程度**: 高/中/低

### 问题描述
15:40 A 股收盘分析未按时推送（应 15:40，现已 15:50）

### 建议操作
请邦德立即检查：
1. cron 任务是否正常
2. 推送脚本是否执行
3. 日志是否有错误

---
钱多多 💰
```

## 🌏 第三章：跨地域通信（新加坡 ↔ 上海）

### 通信挑战

- ❌ **不同 OpenClaw 实例** - 新加坡和上海是独立的 OpenClaw 部署
- ❌ **无法直接会话通信** - `sessions_send` 仅限同一实例
- ❌ **网络环境差异** - 上海网络受限，新加坡网络自由
- ✅ **GitHub 中转** - 通过 GitHub 仓库实现异步通信

### 通信机制

#### 方式 1: GitHub 中转 ⭐（推荐）

**架构**:
```
新加坡节点
    ↓
写入 github-sync 目录
    ↓
git push 到 GitHub
    ↓
上海节点 git pull
    ↓
读取共享文件
```

**共享目录**: `/home/admin/.openclaw/workspace/github-sync/shared/`

**文件约定**:

| 文件 | 用途 | 写入者 | 读取者 |
|------|------|--------|--------|
| `singapore_to_shanghai.md` | 新加坡→上海 | 新加坡代理 | 上海代理 |
| `shanghai_to_singapore.md` | 上海→新加坡 | 上海代理 | 新加坡代理 |
| `daily_sync_report.md` | 每日同步报告 | 新加坡 | 上海 |
| `urgent_notification.md` | 紧急通知 | 双方 | 双方 |

**使用示例**:

```bash
# 新加坡→上海（邦德执行）
cat > /home/admin/.openclaw/workspace/github-sync/shared/singapore_to_shanghai.md << EOF
## 📝 新加坡→上海 | $(date '+%Y-%m-%d %H:%M')

上海同事们好！

今日同步内容:
1. ✅ Skill 安全扫描机制（3 层防护）
2. ✅ 推送系统重构（解决 sub-agent 问题）
3. ✅ 早间报告新标准（GitHub 全网搜索）

推送状态:
- 00:05 服务器报告 ✅
- 08:00 早间新闻 ✅
- 09:00 外汇+A 股 ✅

请上海站点执行:
git pull origin master

---
詹姆斯邦德 🕵️‍♂️ | 新加坡节点
EOF

# 推送到 GitHub
cd /home/admin/.openclaw/workspace/github-sync
git add shared/
git commit -m "📝 新加坡→上海 每日同步 $(date '+%Y-%m-%d')"
git push origin master

# 上海站点拉取
cd /path/to/shanghai-workspace
git pull origin master
cat shared/singapore_to_shanghai.md
```

## 🛡️ 第五章：安全与审计

### 技能安全扫描（新加坡负责）

**3 层防护体系**:

| 层级 | 工具 | 检测内容 | 状态 |
|------|------|----------|------|
| 第 1 层 | VirusTotal | 恶意代码（ClawHub 集成） | ✅ 可用 |
| 第 2 层 | 本地审计 | Prompt Injection/危险命令 | ✅ 可用 |
| 第 3 层 | 人工审查 | 代码审查/权限验证 | ✅ 可用 |

**流程**:
```
发现技能
    ↓
安全扫描（3 层）
    ↓
❌ 未通过 → 终止，禁止安装
✅ 通过 → 生成报告
    ↓
通过 GitHub 同步到上海
    ↓
上海站点安装使用
```

### 推送质量审计（钱多多负责）

**审计频率**: 每天 2 次（08:30 + 21:00）

**审计内容**:
- 推送及时性（误差<5 分钟）
- 推送完整性（无缺失）
- 金融数据准确性（与权威源一致）
- 格式正确性（Markdown 无乱码）
